# BIM Document Manager — полный разбор проекта

> Легитимная дипломная система управления проектной информацией на основе BIM/IFC.

---

## 1. Что это такое

**BIM Document Manager** — это веб-приложение (АСУ проектной информации), которое автоматизирует работу с **IFC-файлами** — открытым стандартом обмена BIM-моделями зданий (ISO 16739). Система превращает «чёрный ящик» IFC-модели в структурированные данные, с которыми можно работать как с базой данных: искать, фильтровать, проверять качество, выгружать отчёты.

**Целевая аудитория:**
- BIM-менеджеры и BIM-координаторы
- Проектировщики (архитекторы, конструкторы, инженеры ОВ/ВК/ЭОМ)
- Технадзор и эксперты, принимающие проектную документацию
- Заказчики, проверяющие комплектность модели

## 2. Какие проблемы решает

### Проблема 1 — Разрозненность данных в BIM-моделях
В IFC-файлах информация хранится в виде графа сотен тысяч сущностей и отношений. Проектировщик в Revit/ArchiCAD видит 3D-модель, но **менеджер/заказчик не может просто открыть IFC и получить список всех стен с площадями** без специализированного ПО (Navisworks, Solibri — дорогие).

**Решение:** система парсит IFC один раз, раскладывает данные по нормализованной БД (проекты → здания → этажи → помещения → элементы) и даёт к ним SQL-подобный доступ через веб-интерфейс.

### Проблема 2 — Качество информационной модели (Information Quality)
Проектные команды сдают IFC с массой ошибок:
- у элементов нет имён, типов, материалов
- стены «висят в воздухе» — не привязаны к этажу
- отсутствуют обязательные property sets (например, `Pset_WallCommon.IsExternal`)
- аномальные геометрии (нулевая площадь, отрицательный объём)

Ручной аудит модели на 50 000 элементов — неделя работы.

**Решение:** модуль `quality_checker` автоматически прогоняет 7 встроенных проверок + произвольное число пользовательских правил и генерирует список замечаний (issues).

### Проблема 3 — Отчётность и экспорт
Экспертам и заказчикам нужны реестры элементов, ведомости объёмов, сводные таблицы — в **XLSX/CSV/PDF**, а не в IFC.

**Решение:** модуль `export_service` + `report_builder` формирует отчёты в привычных форматах.

### Проблема 4 — Визуальная проверка на уровне модели
Открыть IFC локально — нужен Navisworks/BIMvision.

**Решение:** встроенный 3D-просмотрщик на **Three.js + web-ifc** работает прямо в браузере, без установки софта.

### Проблема 5 — Отсутствие системы замечаний
BCF (BIM Collaboration Format) — стандарт сложный, внедряется редко. Проектировщики шлют замечания в почте/Excel → теряются.

**Решение:** встроенный модуль `issues` хранит замечания, привязывает их к конкретным элементам, отслеживает статусы (open / in_progress / resolved / ignored).

## 4. Модель данных (ORM)

Семь сущностей, связанных иерархически:

| Таблица | Назначение | Ключевые поля |
|---|---|---|
| **projects** | загруженная IFC-модель | `name`, `file_path`, `file_size`, `ifc_schema` (IFC2X3/IFC4) |
| **buildings** | здания (из IFC) | `global_id` (GUID из IFC), `name` |
| **storeys** | этажи | `name`, `elevation` (отметка) |
| **spaces** | помещения | `area`, `volume`, `long_name` |
| **elements** | все строительные элементы (стены, двери, окна, плиты, колонны, балки, лестницы, оборудование ОВ/ВК) | `ifc_class`, `material`, количественные характеристики (L/W/H/area/volume/weight), флаги качества, `properties_json` |
| **issues** | замечания | `severity` (error/warning/info), `status`, `category`, `recommendation` |
| **qc_rules** | настраиваемые правила QC | `check_type`, `check_config` (JSON), `is_active` |

Каскадное удаление (`ondelete="CASCADE"`) гарантирует: удалил проект → снёс все дочерние данные.

## 5. Функционал — по модулям

### 5.1. Загрузка и парсинг IFC
- Принимает файл через `POST /api/models/upload`
- `ifc_parser.py` через **ifcopenshell** проходит по модели и извлекает:
  - **Пространственную структуру** (IfcBuilding → IfcBuildingStorey → IfcSpace)
  - **Элементы** 18 классов IFC: `IfcWall`, `IfcWallStandardCase`, `IfcSlab`, `IfcRoof`, `IfcColumn`, `IfcBeam`, `IfcDoor`, `IfcWindow`, `IfcStair`, `IfcRailing`, `IfcCurtainWall`, `IfcPlate`, `IfcMember`, `IfcCovering`, `IfcFurnishingElement`, `IfcBuildingElementProxy`, `IfcOpeningElement`, `IfcFlowTerminal/Segment/Fitting`, `IfcDistributionElement`
  - **Геометрические характеристики** (BaseQuantities): Length, Width, Height, Area, Volume, Weight
  - **Материалы** (single material, MaterialLayerSet, MaterialList)
  - **Property Sets** — все пользовательские параметры сохраняются в `properties_json`
  - **Принадлежность этажу** через `IsContainedInStructure`

### 5.2. Пространственная структура
- `GET /api/models/{id}/structure` → дерево здание → этаж → помещение
- Frontend (`StructureTab`) — раскрывающееся дерево с указанием площадей помещений

### 5.3. Таблица элементов
- `GET /api/models/{id}/elements?ifc_class=...&storey=...&search=...`
- Фильтрация: по классу, этажу, материалу, флагу «проблемный»
- Поиск по имени / global_id
- Клик по строке → переход в 3D-просмотрщик с подсветкой элемента

### 5.4. 3D-просмотрщик
- Рендеринг прямо в браузере через **web-ifc** (WASM-версия IFC-парсера) + **Three.js**
- Подсветка проблемных элементов, вращение, замеры
- Работает **клиент-сайд** — не грузит сервер

### 5.5. Контроль качества (ядро системы)

**Встроенные проверки (`check_element`):**

| # | Категория | Что проверяет | Severity |
|---|---|---|---|
| 1 | `missing_name` | у элемента нет `Name` | warning |
| 2 | `missing_type` | нет типоразмера (IfcTypeObject) | warning |
| 3 | `missing_storey` | элемент не привязан к этажу | **error** |
| 4 | `missing_material` | нет материала (только для несущих и ограждающих) | warning |
| 5 | `missing_quantities` | нет количественных характеристик | **error** |
| 6 | `anomaly_area` | подозрительно малая площадь (< 0.01 м²) | warning |
| 7 | `anomaly_volume` | отрицательный объём | **error** |

**Пользовательские правила (`qc_rules`)** — 4 типа:
- `required_property` — обязательное свойство в Pset (например, `Pset_WallCommon.IsExternal`)
- `value_range` — значение в диапазоне (например, `area ∈ [0.1, 10000]`)
- `has_quantity` — наличие конкретной количественной характеристики
- `has_storey` — привязка к этажу

Правила активируются/отключаются флагом `is_active`, могут применяться ко всем классам (`*`) или к конкретному.

### 5.6. Детектор аномалий
Файл `anomaly_detector.py` — дополнительный слой статистического анализа (outlier detection), дополняющий формальные правила.

### 5.7. Замечания (Issues)
- Генерируются автоматически по результатам QC + создаются вручную
- Жизненный цикл: `open → in_progress → resolved / ignored`
- Привязка к элементу (с каскадом)
- Категоризация и рекомендации по исправлению (модуль `recommendation.py`)

### 5.8. Аналитика
- Дашборд с графиками через **Recharts**:
  - распределение элементов по классам (pie chart)
  - количество элементов по этажам (bar chart)
  - % полноты данных (прогресс-бары)
  - динамика замечаний по категориям и severity

### 5.9. Экспорт отчётов
- **XLSX** — реестр элементов, список замечаний, ведомости объёмов
- **CSV** — для интеграции с Excel/1С
- **PDF** — формализованный отчёт для заказчика

## 7. Сильные стороны проекта

1. **Правильная асинхронная архитектура** — FastAPI + async SQLAlchemy + asyncpg корректно работают под нагрузкой.
2. **Чистое разделение слоёв** — `api/routes` → `services` → `models` (классический DDD-подход).
3. **Расширяемость** — пользовательские QC-правила добавляются без деплоя кода.
4. **Безопасность** — в `_apply_custom_rule` есть `ALLOWED_ATTRIBUTES` whitelist, защищающий `getattr()` от инъекций.
5. **One-command-deploy** — `docker compose up` поднимает всё.
6. **Производительность фронта** — web-ifc парсит модель в браузере через WASM, сервер не нагружается рендером.
7. **Каскадные удаления** в БД — нет «висячих» записей.

## 9. Итог

Проект представляет собой **полнофункциональный BIM Model Checker** с элементами issue-трекера и аналитической панели. По функционалу он покрывает значительную часть задач, которые решают коммерческие системы уровня Solibri Office / BIMcollab ZOOM, при этом имеет современную архитектуру (async FastAPI, React, WASM-рендеринг 3D) и готов к промышленной эксплуатации благодаря Docker-контейнеризации.
